# KB daily contextual reindex

## Objective

Keep the local knowledge-base FAISS index fresh **and contextual**: newly added
or edited notes get per-chunk prefaces (RFC 017) prepended for embedding, which
improves retrieval. `kb doctor` reports modified/new ingestable files that are
not yet indexed; every one is content `kb search` cannot return. This playbook
runs a daily incremental contextual reindex to clear that.

This is an operational maintenance task. **Do not** edit source code, change git
state, or open PRs. The only write is the index rebuild via the `kb` CLI.

## Environment

The scheduled agent may not inherit the interactive shell environment. The kb CLI
auto-loads the repo-root `.env`, which supplies the LLM config:

| Var | Value | Notes |
|---|---|---|
| `KB_LLM_PROVIDER` | *(absent)* | Absent ⇒ resolves to **`local`**. See the warning below. |
| `KB_LLM_ENDPOINT` | `http://127.0.0.1:11434/v1/chat/completions` | Ollama's OpenAI-compatible route |
| `KB_LLM_MODEL` | `qwen3:4b-instruct-2507-q4_K_M` | The preface model |
| `KB_OPENROUTER_API_KEY` | *(set, but unused)* | Inert while the provider is `local` |
| `KB_CONTEXTUAL_RETRIEVAL` | `on` | |
| `KB_CONTEXTUAL_CONCURRENCY` | `4` | |

> **Prefaces are generated by a LOCAL model (qwen3:4b via Ollama) — not DeepSeek,
> not OpenRouter.** An earlier version of this playbook claimed OpenRouter/DeepSeek
> at concurrency 10. That was never true of this machine: every one of the ~4,800
> preface sidecars on disk records `"model": "qwen3:4b-instruct-2507-q4_K_M"`, and
> zero record DeepSeek. `resolveLlmProvider` (`src/config/llm-provider.ts:63`)
> defaults to `local` unless `KB_LLM_PROVIDER` is *exactly* `openrouter`, and that
> var is not in `.env`. Do not "fix" this by setting it — see
> [Changing the preface model](#changing-the-preface-model-read-first) first.

Ollama serves **both** models on `:11434` — `nomic-embed-text` (embeddings) and
`qwen3:4b` (chat/prefaces). Nothing needs to be started by hand; do **not** start
a separate `llama-server`.

Export the path/index vars explicitly:

```bash
export KNOWLEDGE_BASES_ROOT_DIR="$HOME/knowledge_bases"
export FAISS_INDEX_PATH="$HOME/knowledge_bases/.faiss"
export OLLAMA_MODEL=nomic-embed-text:latest
export EMBEDDING_PROVIDER=ollama
export OLLAMA_BASE_URL=http://localhost:11434
export LOG_FILE="$HOME/knowledge_bases/.kb-canonical.jsonl"
export KB_LOG_FORMAT=both
export PATH="$HOME/.nvm/versions/node/v24.11.1/bin:$PATH"
```

## Steps

1. **Preflight the LLM — and check *which* LLM.**

   ```bash
   kb llm probe
   ```

   `chat_ok: true` on its own is **not** a sufficient gate. It only proves that
   *something* answered at the configured endpoint — it does not tell you which
   provider or model resolved. That is exactly how the DeepSeek fiction above
   survived: the probe went green against local Qwen while the runbook claimed
   DeepSeek. Assert the endpoint and model explicitly:

   ```bash
   kb llm probe | grep -q '127.0.0.1:11434' && \
     grep -q '^KB_LLM_MODEL=qwen3:4b' .env && echo "provider OK: local qwen3:4b"
   ```

   If the endpoint or model is not what you expect, **stop and report** — do not
   reindex. A preface written by the wrong model is currently permanent (see
   below). Expected probe output:

   ```json
   {
     "endpoint": "http://127.0.0.1:11434/v1/chat/completions",
     "health_ok": false,
     "chat_ok": true,
     "detail": "chat completion succeeded; health HTTP 404"
   }
   ```

   `health_ok: false` / `health HTTP 404` is **normal and expected** — Ollama has
   no `/health` route. `chat_ok: true` is the signal that matters.

   (Issue #861 tracks making `kb llm probe` print the resolved provider/model and
   adding `--expect-provider`, which would replace this manual grep.)

   If the LLM is genuinely unreachable, the reindex still rebuilds the index but
   prefaces become retry-pending; note it and continue.

2. **Assess.** `kb doctor 2>&1 | grep -i stale` — record the modified/new count.
   If it reports "no modified or new ingestable files detected," there is nothing
   to do; report and stop.

3. **Run the contextual reindex (respect the research-agent window).**

   ```bash
   kb reindex --with-context
   ```

   - Cache-aware: only chunks whose contextual-preface sidecar is missing/invalid
     call the LLM. In steady state most runs make **zero** LLM calls (`cold=0`,
     everything a cache hit) — that is success, not a skipped step.
   - Prefaces are issued at `KB_CONTEXTUAL_CONCURRENCY` (**4**).
   - **Expect a full FAISS rebuild.** Any changed file whose chunk manifest isn't
     a stable append forces a global rebuild, because FAISS vector deletion is
     unsupported. So the run re-embeds the whole corpus (~32.5k chunks, ~8–12 min)
     even when no LLM call is made. This is normal; the log says so explicitly
     ("Rebuilding the full FAISS index because at least one changed file could not
     be updated incrementally"). Budget wall-clock for the rebuild, not the LLM.
   - **Do NOT pass `--force`.** `kb reindex` refuses to start inside the
     research-agent window (06:00–10:30 UTC) or if its estimated runtime would
     cross into it. If it *is* refused, **report it and stop** — do not force into
     the research window; the next run catches up.

4. **Verify.** Re-run `kb doctor 2>&1 | grep -i stale`; confirm it reports
   "no modified or new ingestable files detected". Sanity-check the index is
   queryable with a real search, e.g. `kb search "contextual retrieval preface"`.

5. **Report.** Staleness before/after, preface coverage / failed / retry-pending
   counts, and elapsed time.

## Reading the run summary

A healthy steady-state run looks like this (real output, 2026-07-13):

```
  contextual_estimate:   cold=0 cache_hits=32012 retry_skips=476
  took_ms:               706698
  summary: files_scanned=4833 files_changed=684 files_unchanged=4149 warnings=0 failures=0
  contextual: covered=32237 failed=476 retry_pending=476 errors=[truncated_doc=476]
```

Two lines routinely get misread as regressions:

- **`cold=0`** — no LLM calls were made. Correct and expected when every changed
  chunk already had a valid cached preface. It does not mean prefacing was skipped.
- **`failed=476 retry_pending=476 errors=[truncated_doc=476]`** — **not a failure
  to fix.** `truncated_doc` is a *by-design, non-retryable* outcome: in oversized
  documents the LLM only sees a leading prefix, so chunks past the truncation
  limit cannot be situated (`src/contextual-preface.ts:195`). Its backoff is
  `POSITIVE_INFINITY` — "never until the file changes"
  (`src/config/contextual-preface.ts:69`). Despite the `retry_pending` label,
  nothing will retry these. The chunks are still indexed and searchable, just
  without a preface. Expect this count to persist run over run; only investigate
  if it *grows sharply* or if other error kinds appear alongside it.

## Changing the preface model — read first

**Do not switch the preface model (e.g. to OpenRouter/DeepSeek) as part of this
nightly job.** The preface cache does not include `model` in its validity check
(`src/contextual-preface.ts:163-171`), despite a header comment claiming it does.
Consequence: whichever model wrote a preface owns that chunk **permanently**. A
model switch would convert only the chunks of files that happen to be stale that
night and silently leave every unchanged file on the old model — a mixed-model
corpus with no signal that it happened.

That is tracked as **issue #860**. A real model migration requires #860 fixed (or
a `GENERATOR_VERSION` bump), followed by a deliberate full re-preface of ~32.5k
chunks. That is a scheduled migration, not a nightly incremental.

## Success criteria

The contextual reindex completed (staleness cleared) or was deliberately deferred
for the LRA-window guard with a clear report. The resolved LLM provider was
verified to be the intended one before any preface was written. No source files or
git state were touched.
